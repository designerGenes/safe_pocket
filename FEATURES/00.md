## What is Safe_Pocket about?

Safe_Pocket is about making ad hoc VS code workspaces, which are defined by the directories contained, rather than necessarily by a single project name.  For exmample, in iOS (which has a very limited scope), you often never work outside of XCode and a single project or several projects in a monorepo.  Outside of iOS, you will often have need to keep in context a project which could be an iOS project, as well as multiple other local directories.  These clusters of things often have project names they already belong to, so naming a new project based on the clusters together doesn't work.  

Instead, Safe_Pocket will generate a "safe pocket" folder maintained (and git versioned) locally (remotely if preferred), based on an intelligent hash made from the file identities of the directories (clusters) involved in this generated workspace file.  This gives you several benefits, in particular in Enterprise situations.

To understand safe_pocket, you must grasp two important concepts:
- it is difficult (friction-inducing) to get code merged into other people's codebases
- if you know how to customize AI, it can serve you many times more than if uncustomized or unused.

If your codebase is like this

- Monorepo
- - Feature1
- - Feature2
- - - Subfeature
- - FeatureYouAreWorkingOn

There may be value to you, specifically, in giving instructions that you have found useful in general to copilot.  You can submit a PR for your feature.  It can get approved.  You move to another feature.

Oh no, your useful instructions are gone.  No worry, you can make another PR to get your code into this feature.  It can approved.  You can do this forever, over and over.  But soon you'll need to update your instructions, and you can't do that across all those subfeatures every time.  You need your code in the monorepo.  And that's hard to get done.

So what safe_pocket does is lets you skip all of that.  The "code" of your copilot instructions is only relevant to copilot because copilot has been told to look for it in a particular "project file" location.  safe_pocket generates this location by sorting, hashing and combining the file paths of the folders "involved" in the workspace it has generated.  Copilot then knows to look in this folder (which is git maintained and can optionally include out of the box installations of things like Beads) for instructions, "Features" (markdown documents of loose-form ideas for the next features to be implemented), "Observations" (markdown files generated automatically by Copilot after it has made what it considers a significant discovery, for better or worse, to improve itself and be read before the start of each session)

safe_pocket (aliased to spocket) should have a very simple user interface.  We will build it in Rust.  The UI will be gently colored.

spocket --help
spocket --list   <-- (print all spocket "aliases")
spocket --register foo="$(pwd)" <-- (register 'foo' as an alias for the current directory)
spocket --register bar="$HOME/dev/bar". <-- (register 'bar' as an alias for the expanded "$HOME/dev/bar")
spocket -i foo    <-- either create new workspace if 'foo' was registered but never opened, or open workspace made by hashing the file path of the path "aliased" to foo.  Note this is an 'internal' alias to safe_pocket, not a system alias.
spocket -i $platypus  <-- as above, but this makes or reuses a workspace from an expected external alias belonging to key "platypus"
spocket --unregister foo
spocket -i bar -i $platypus -i /users/duck/dev/  <-- new workspace from all three locations, even though some belong already to other workspaces.  The name of the "safe pocket" directory of this workspace is made by sorting the full paths of these combined locations and hashing them.  Then we create or read from a directory with this name.  The safe_pocket directory is locally maintained automatically, but we should have a way to commit it your own github repo if preferred

It would be very unlikely that you could get Copilot prompts or instructions specific to this context  merged into the monorepo.  Why would anyone allow that in?  It has no use... except to you.  It has tremendous use to you, especially if you apply creative work to a lot of the same situations iteratively, or recurringly.  It gives you the power to make macros and automations of your job.  So it is free software that rather than downloading, you just need to build.

when I call spocket, I want it to look like this

```bash
alias devBin="$HOME/dev/bin"
spocket -i $devBin -i $HOME/dev/personal_project --with-beads

```

1. spocket sorts and hashes the full paths of both of these folders together into a "pocket name"
2. spocket then checks it private internal safe pockets directory for a safe pocket with this pocket name
3. if none is found, a new one is created.  It looks roughly like this

.spocket
-- b4499X11f/ (this is a pretend version of the hashed name)
--- .github/
---- prompts/
---- copilot-instructions.md
--- FEATURES/
---- 00.md (empty for now if nothing already here)
--- observations/
--- b4499X11f.workspace (or whichever extension is used by VS Code workspaces)
--- (some folders specific to setting up beads)

It will create or reload a vs code workspace with these folders

- $HOME/dev/bin
- $HOME/dev/personal_project
- $HOME/spocket/b4499X11f (the hashed name of the involved folder paths)

Now, this is not entirely perfect.  Sometimes you have to drag the associated copilot-instructions into the copilot window to get it to become recognized.  But it will be recognized even if you must reload the VS Code window.

The goal of safe_pocket is to extend copilot and make its interfacing with vs code workspaces as efficient as possible, and access the customizability of the AI agent without needing to get approval to commit code that makes you stronger into someone else's codebase

Safe_pocket will eventually grow to fill these needs:
1. editing an existing workspace by adding directories, without losing associated files (if the hashed name of the files change, we might sever the connection to their safe pocket folder by accident)
2. backwards compatability will be a struggle
3. we want to be able to customize the safe pocket innards quite a lot, even if not right away.  For example, I would like to be able to hook a safe pocket to a github repo (or create a new gitub repo and then hook it to this safe pocket) at initialization.  Maybe I want to install a Ralph Wiggum loop to my API interactions or include some previously saved custom prompts from somewhere else into my generated workspace
